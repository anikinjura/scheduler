# 🔧 Документация парсера отчетов Ozon ПВЗ

## 📋 Общее описание

Парсер отчетов — это модульная система для извлечения данных из веб-приложения **Ozon ПВЗ** с использованием Selenium WebDriver. Архитектура построена на принципах объектно-ориентированного программирования с использованием абстрактных классов и наследования, что позволяет легко расширять функциональность для различных задач.

**Назначение:**
- Автоматический сбор данных из системы управления ПВЗ Ozon
- Поддержка многошаговой обработки данных
- Работа с сохранённой сессией пользователя (без повторной авторизации)
- Проверка и установка правильного ПВЗ
- Закрытие информационных оверлеев
- Сохранение отчётов в формате JSON
- Интеграция с Google Sheets и Telegram (через scheduler_runner)

#### BaseParser

**Файл:** `core/base_parser.py`

Базовый класс, предоставляющий общую функциональность для работы с веб-браузером Edge через Selenium:

**Основные возможности:**
- Управление сессией браузера (настройка, закрытие, завершение процессов)
- Поддержка существующей пользовательской сессии (работа с профилем Edge)
- Универсальные методы для взаимодействия с элементами страницы
- Поддержка различных типов элементов: `input`, `select`, `checkbox`, `radio`, `dropdown`
- Механизмы извлечения данных из таблиц (стандартные и динамические)
- Обработка выпадающих списков
- Детальное логирование всех операций
- Очистка Lock-файлов профиля браузера

**Абстрактные методы (требуют реализации в дочерних классах):**
- `login()` - реализация логики входа в систему
- `navigate_to_target()` - реализация навигации к целевой странице
- `extract_data()` - реализация извлечения данных
- `logout()` - реализация выхода из системы

**Ключевые методы:**
- `setup_browser()` - настройка веб-браузера с использованием профиля пользователя
- `close_browser()` - закрытие браузера и завершение процессов
- `get_element_value()` - получение значения элемента (текст, атрибут, значение input)
- `set_element_value()` - установка значения элемента (включая dropdown)
- `extract_table_data()` - извлечение данных из таблицы по конфигурации
- `_select_option_from_dropdown()` - выбор опции из выпадающего списка
- `_terminate_browser_processes()` - завершение процессов браузера перед запуском
- `_cleanup_lock_files()` - удаление Lock-файлов из профиля
- `_get_default_browser_user_data_dir()` - получение пути к профилю пользователя
- `_get_current_user()` - получение имени текущего пользователя

**Конфигурация:**
- `DEFAULT_TIMEOUT` - таймаут ожидания элементов (60 сек)
- `BROWSER_EXECUTABLE` - имя исполняемого файла браузера ('msedge.exe')
- `EDGE_USER_DATA_DIR` - путь к пользовательским данным Edge
- `HEADLESS` - режим headless для браузера (False для стабильности)
- `table_configs` - конфигурации для извлечения данных из таблиц

#### BaseReportParser

**Файл:** `core/base_report_parser.py`

Класс, расширяющий `BaseParser` с функциональностью для работы с отчётами:

**Основные возможности:**
- Поддержка мульти-шаговой обработки данных
- Форматирование и сохранение отчётов в различных форматах (JSON, CSV, XML)
- Управление конфигурацией отчётов
- Валидация данных отчёта
- Поддержка аргументов командной строки
- Формирование URL с фильтрами по дате и типу данных
- Автоматическое проставление временных меток

**Абстрактные методы (требуют реализации в дочерних классах):**
- `get_report_type()` - возвращает тип отчёта (например, 'multi_step_ozon_report')
- `extract_report_data()` - извлекает специфичные данные отчёта

**Ключевые методы:**
- `run_parser()` - запуск парсера отчётов (переопределённый метод)
- `format_report_output()` - форматирование данных отчёта
- `save_report()` - сохранение отчёта в файл
- `_execute_multi_step_processing()` - выполнение мульти-шаговой обработки
- `_execute_single_step()` - выполнение одного шага обработки
- `_build_url_filter()` - формирование фильтра для URL
- `_update_config_for_step()` - обновление конфигурации для шага
- `_combine_step_results()` - комбинирование результатов шагов
- `_aggregate_nested_results()` - агрегация вложенных результатов
- `_apply_post_processing()` - применение пост-обработки данных
- `get_common_report_info()` - получение общей информации отчёта
- `get_common_url_prefix()` - получение общего URL для отчётов

**Мульти-шаговая обработка:**
Поддерживает сложные сценарии извлечения данных через конфигурацию `multi_step_config`:
```python
"multi_step_config": {
    "steps": [
        {
            "step_id": "giveout",
            "data_type_filter": "giveout",
            "url_filter_template": "...",
            "extraction_mode": "table"
        },
        {
            "step_id": "direct_flow",
            "data_type_filter": "direct",
            "extraction_mode": "standard"
        }
    ]
}
```

#### OzonReportParser

**Файл:** `core/ozon_report_parser.py`

Класс, расширяющий `BaseReportParser` с функциональностью для работы с системой Ozon ПВЗ:

**Основные возможности:**
- Проверка и установка правильного ПВЗ (location_id)
- Обеспечение корректной работы с конкретным пунктом выдачи
- Интеграция с логикой навигации для проверки ПВЗ
- **Автоматическое закрытие информационных оверлеев** (модальных окон Ozon)
- Проверка активного backdrop (полупрозрачного фона)

**Ключевые методы:**
- `get_current_pvz()` - извлечение текущего ПВЗ с страницы
- `set_pvz(target_pvz)` - установка нужного ПВЗ в выпадающем списке
- `ensure_correct_pvz()` - проверка и установка правильного ПВЗ
- `navigate_to_target()` - переопределён для проверки ПВЗ после навигации
- `build_url_filter()` - вспомогательный метод для формирования фильтра URL
- `_check_and_close_overlay()` - проверка и закрытие оверлея
- `_is_overlay_present()` - проверка наличия оверлея на странице
- `_is_backdrop_active()` - проверка активного backdrop
- `_click_close_button(selector)` - клик по кнопке закрытия оверлея

**Конфигурация оверлеев (`overlay_config`):**
```python
"overlay_config": {
    "enabled": True,
    "overlay_selector": "//div[contains(@class, 'ozi__dialog__dialog__C2BB8')]",
    "close_button_selector": "//button[contains(@class, 'ozi__button') and normalize-space()='Отложить']",
    "wait_timeout": 5,
    "retry_count": 3,
    "retry_delay": 1,
    "backdrop_selectors": [
        "//div[contains(@class, 'ozi__backdrop__backdrop__')]",
        "//div[contains(@class, 'ozi__backdrop')]",
        "//div[contains(@class, 'backdrop')]",
        "//div[contains(@class, 'modal-backdrop')]"
    ]
}
```

**Алгоритм работы с оверлеями:**
1. Проверка включения (`enabled=True`)
2. Проверка наличия оверлея по `overlay_selector` (таймаут `wait_timeout`)
3. Если оверлей обнаружен:
   - Клик по кнопке закрытия (`close_button_selector`)
   - Проверка исчезновения оверлея
   - Повтор до `retry_count` раз с задержкой `retry_delay`
4. Проверка активного backdrop (таймаут 1 сек на селектор)

**Места вызова проверки оверлеев:**
- Перед открытием dropdown при установке ПВЗ
- После выбора ПВЗ из dropdown
- Перед проверкой текущего ПВЗ в `ensure_correct_pvz()`

#### MultiStepOzonParser

**Файл:** `implementations/multi_step_ozon_parser.py`

Конкретная реализация парсера для сбора данных из Ozon ПВЗ:

**Назначение:**
- Сбор данных по выдачам (giveout)
- Сбор данных по прямому потоку (Direct flow)
- Сбор данных по возвратному потоку (Return flow)
- Агрегация результатов в единый отчёт

**Этапы обработки:**
1. **Giveout** - подотчёт по выдачам (количество выданных заказов)
2. **Direct flow** - подотчёт прямому потоку (принятые заказы)
3. **Return flow** - подотчёт возвратному потоку (возвращённые заказы)
4. **Summary** - агрегация всех данных с итоговой статистикой

**Особенности реализации:**
- Использует сохранённую сессию браузера (без авторизации)
- Не требует явного выхода из системы
- Автоматически очищает служебные поля из результатов
- Поддерживает извлечение данных из таблиц
- Формирует итоговый отчёт в формате JSON


### Мульти-шаговая обработка (MultiStepOzonParser)

```
Шаг 1: Giveout (выдачи)
  ├── Переход на страницу отчёта по выдачам
  ├── Извлечение таблицы с данными
  └── Сохранение результата: {giveout: N}

Шаг 2: Direct flow (прямой поток)
  ├── Переход на страницу отчёта по прямому потоку
  ├── Извлечение данных
  └── Сохранение результата: {direct_flow_total: N}

Шаг 3: Return flow (возвратный поток)
  ├── Переход на страницу отчёта по возвратному потоку
  ├── Извлечение данных
  └── Сохранение результата: {return_flow_total: N}

Шаг 4: Summary (итоговый)
  ├── Агрегация всех данных
  ├── Добавление временной метки
  ├── Добавление информации о ПВЗ
  └── Формирование финального отчёта
```

## 🛠️ Работа с оверлеями

### Проблема

На странице Ozon ПВЗ присутствуют два типа оверлеев:
1. **Большой оверлей** "Новости Турбо ПВЗ" с кнопкой "Отложить" и крестиком
2. **Маленький оверлей** уведомлений "У вас 9 непрочитанных сообщений" с кнопкой "Перейти"

Маленький оверлей перекрывает крестик большого оверлея, поэтому клик по крестику не проходит.

### Решение

Используется кнопка "Отложить" вместо крестика:
- Селектор: `//button[contains(@class, 'ozi__button') and normalize-space()='Отложить']`
- Метод: Selenium клик (без JavaScript)
- Проверка через `WebDriverWait` с таймаутом 5 секунд

### Алгоритм закрытия оверлея

```python
def _check_and_close_overlay(self) -> bool:
    # 1. Проверка включения (overlay_config.enabled)
    # 2. Проверка наличия оверлея (overlay_selector, timeout=5s)
    # 3. Если оверлей найден:
    #    3.1. Клик по кнопке закрытия (close_button_selector)
    #    3.2. Проверка исчезновения оверлея
    #    3.3. Повтор до retry_count раз с retry_delay
    # 4. Проверка активного backdrop (4 селектора, timeout=1s каждый)
    # 5. Возврат True (оверлея нет) или False (не удалось закрыть)
```

### Диагностика оверлеев

Для тестирования селекторов создан микросервис `overlay_closer.py`:

```bash
# Тестирование всех селекторов кнопки "Отложить"
python overlay_closer/overlay_closer.py --strategy all

# Тестирование одного селектора
python overlay_closer/overlay_closer.py --strategy postpone
```

Скрипт:
- Открывает браузер с профилем пользователя
- Переходит на страницу отчётов
- Тестирует каждый селектор
- Выводит результат: найден/не найден, кликабелен/не кликабелен

## ⚙️ Конфигурация

Парсер использует гибкую систему конфигурации через словарь:

### Основные разделы конфигурации

1. **Параметры браузера:**
   - `BROWSER_EXECUTABLE` - имя исполняемого файла ('msedge.exe')
   - `EDGE_USER_DATA_DIR` - путь к профилю пользователя
   - `HEADLESS` - режим headless (False для стабильности)
   - `DEFAULT_TIMEOUT` - таймаут ожидания элементов (60 сек)

2. **Селекторы элементов:**
   - `selectors.pvz_selectors` - селекторы для работы с ПВЗ
   - `pvz_input` - селектор input ПВЗ
   - `pvz_dropdown` - селектор dropdown
   - `pvz_option` - селектор опции

3. **Параметры URL:**
   - `base_url` - базовый URL системы Ozon
   - `filter_template` - шаблон фильтра для URL
   - `date_filter_template` - шаблон фильтра по дате
   - `data_type_filter_template` - шаблон фильтра по типу данных

4. **Конфигурация таблиц:**
   - `table_configs` - словарь с конфигурациями таблиц
   - `table_selector` - XPath таблицы
   - `table_type` - тип таблицы ('standard', 'dynamic')
   - `table_columns` - описание колонок (name, selector, regex)

5. **Конфигурация оверлеев:**
   - `overlay_config.enabled` - включение проверки
   - `overlay_config.overlay_selector` - селектор оверлея
   - `overlay_config.close_button_selector` - селектор кнопки закрытия
   - `overlay_config.wait_timeout` - таймаут ожидания (5 сек)
   - `overlay_config.retry_count` - количество попыток (3)
   - `overlay_config.backdrop_selectors` - селекторы backdrop

6. **Дополнительные параметры:**
   - `additional_params.location_id` - ID локации (ПВЗ)
   - `report_type` - тип отчёта
   - `execution_date` - дата выполнения

7. **Мульти-шаговая конфигурация:**
   - `multi_step_config.steps` - список шагов обработки
   - `multi_step_output_config` - настройка вывода результатов

## 🔍 Логирование

Парсер использует детальное логирование через кастомный логгер:

**Уровни логирования:**
- `TRACE` - вход/выход из методов
- `DEBUG` - отладочная информация (селекторы, значения, конфигурация)
- `INFO` - информационные сообщения (успешные операции)
- `WARNING` - предупреждения (некритичные ошибки)
- `ERROR` - ошибки (с полным стек трейсом)

**Пример логов:**
```
[10:32:27] TRACE Попали в метод OzonReportParser.__init__
[10:32:28] DEBUG Селекторы ПВЗ: {...}
[10:32:29] INFO Текущий ПВЗ: ТВЗ Москва Люблинская 12
[10:32:30] DEBUG Проверка оверлея: selector=//div[contains...], timeout=5s
[10:32:31] INFO Оверлей найден, закрываем...
[10:32:32] INFO Клик по кнопке закрытия успешно выполнен
```

## 🐛 Отладка и диагностика

### DEBUG_GUIDE.md

Полное руководство по отладке парсера находится в файле [`DEBUG_GUIDE.md`](DEBUG_GUIDE.md).

**Описанные проблемы:**
1. Зависание парсера и сбой цикла выполнения (28.02-01.03.2026)
2. Оверлей не закрывается (03.03.2026)
3. Перенаправление на страницу логина (февраль 2026)

### Типичные ошибки и решения

| Ошибка | Причина | Решение |
|--------|---------|---------|
| `SessionNotCreatedException` | Lock-файл в профиле, headless режим | Завершить процессы браузера, удалить Lock-файлы, отключить headless |
| `element click intercepted` | Элемент перекрыт другим оверлеем | Использовать другой селектор (кнопка вместо крестика) |
| `timeout` при проверке backdrop | Долгий implicit_wait | Использовать WebDriverWait с коротким таймаутом (1 сек) |
| Страница логина вместо отчёта | Истекла сессия | Проверить валидность сессии, обновить при необходимости |

## Update 2026-03-07: anti-fail flow and status model

В коде парсера добавлена fail-fast логика для корректного завершения при проблемах с авторизацией и навигацией.

Ключевые изменения поведения:
- `BaseReportParser._execute_multi_step_processing()` теперь рассчитывает и добавляет служебный флаг `__RUN_STATUS__` (`success`/`partial`/`failed`).
- `BaseReportParser.run_parser()` теперь проверяет `__RUN_STATUS__` и аварийно завершает запуск, если статус `failed`.
- `BaseReportParser._execute_single_step()` при провале навигации с причиной `login_redirect` поднимает `AUTH_REQUIRED: redirected_to_login`.
- `BaseReportParser.navigate_to_target()` сохраняет причину провала в `_last_navigation_failure_reason` и очищает ее при успешной навигации.
- `MultiStepOzonParser.login()` теперь возвращает `False`, если проверка текущей сессии завершилась исключением (раньше такой сценарий мог проходить как успешный).

Важно для анализа логов:
- Если в логе есть `AUTH_REQUIRED:*`, это считается признаком потери валидной сессии и обработка шагов должна завершаться без попытки «дожать» оставшиеся шаги.
- При наличии `__RUN_STATUS__ = failed` результат парсинга не должен считаться успешным даже при частично заполненных данных.
